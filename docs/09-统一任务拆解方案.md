# 主AGENT + 子AGENT + 测试AGENT 工作流

日期：2026-04-13

## 目标

这套工作流的核心目标是提高开发效率：

1. 主 AGENT 负责拆任务和派工
2. 子 AGENT 负责具体实现
3. 测试 AGENT 负责测试和反馈
4. 主 AGENT 负责最终验收和总结

## 三方分工

### 主 AGENT

负责：

1. 理解需求
2. 持续接收新的需求
3. 拆解任务
4. 分配给子 AGENT
5. 判断哪些任务可以并行
6. 汇总所有结果
7. 接收测试 AGENT 反馈
8. 决定接受还是重新优化
9. 通过后再总结给用户

### 子 AGENT

负责：

1. 处理单个开发子任务
2. 在限定 `scope` 中改代码
3. 返回修改说明、验证结果、风险

### 测试 AGENT

负责：

1. 接收主 AGENT 汇总后的结果
2. 跑可用检查
3. 验证功能是否符合预期
4. 输出 PASS/FAIL 和优化建议

## 工作流阶段

1. `Intake And Baseline`
   - 主 AGENT 理解需求并识别项目检查能力
2. `Split And Dispatch`
   - 主 AGENT 切分子任务并派工
3. `Child Implementation`
   - 子 AGENT 并行实现独立任务
4. `Tester Validation`
   - 测试 AGENT 测试并反馈主 AGENT
5. `Main Acceptance Decision`
   - 主 AGENT 验收
   - 正确则进入总结
   - 不通过则回到优化
6. `Finalize And Summarize`
   - 主 AGENT 只总结已验收通过的结果

## 持续接需求

这套工作流支持你在旧任务还没结束时继续提新需求。

主 AGENT 的处理方式是：

1. 为每个新需求生成一个新的 `request_id`
2. 把这个需求拆成独立的一组任务
3. 将新任务追加到现有任务池
4. 根据依赖和 scope 再安排执行顺序

所以它不是“一次只处理一个需求”的模式，而是“持续接单、持续派工”的模式。

## 拆任务原则

主 AGENT 拆任务时按这个顺序：

1. 先按交付目标拆
2. 再按代码边界拆
3. 再按验收方式拆
4. 最后补依赖关系

所以：

1. `dependencies` 只是顺序控制
2. `skills` 只是执行或测试标签
3. 真正决定拆分的是能否独立开发和独立测试

## 这套模式为什么更适合你

因为你要的不是一个“自己全做完”的代理，而是一个“主控调度 + 多子任务并行 + 测试回流 + 主控收口”的开发工作流。

这套就是按这个目标收敛出来的。

## 借鉴来源怎么落进来的

1. 来自 `karpathy/autoresearch`
   - 借了“执行 -> 测试 -> 决策 -> 回到优化”的循环
2. 来自 `msitarzewski/agency-agents`
   - 借了主控 orchestration + 专项 agent 派工
3. 来自 `affaan-m/everything-claude-code`
   - 借了验证优先、质量门、Codex 资产化组织方式
