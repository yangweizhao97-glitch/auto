# 自动化工作流测通需求文档（草案 v0.1）

日期：2026-04-13

## 1. 目标

1. 验证一条完整自动化链路：任务拆分 -> 依赖编排 -> 子代理执行 -> 技能验收 -> 状态回写。
2. 确保流程可重复、可追踪、可扩展。

## 2. 范围

1. `AGENT` 规范文档接入与校验（按规范执行）。
2. 任务 `json` 文件管理（你提到的 `ison` 这里按 `json` 理解）。
3. 子代理调度与上下文代码下发。
4. 基于 `skills` 的测试验收与结果判定。

## 3. 核心数据结构（json）

每个任务必须包含以下字段：

- `id`：唯一标识
- `description`：任务描述
- `status`：状态（`todo` / `in_progress` / `done` / `blocked`）
- `dependencies`：依赖任务 `id` 列表

示例：

```json
{
  "tasks": [
    {
      "id": "task_001",
      "description": "解析需求并切割子任务",
      "status": "todo",
      "dependencies": []
    },
    {
      "id": "task_002",
      "description": "按依赖顺序执行子代理",
      "status": "todo",
      "dependencies": ["task_001"]
    }
  ]
}
```

## 4. 功能需求

1. 任务切割：支持将大任务拆分为多个原子任务并写入 `json`。
2. 依赖管理：执行前检查依赖是否满足；未满足则阻塞。
3. 状态流转：自动维护 `todo -> in_progress -> done/blocked`。
4. 子代理执行：主代理可按任务粒度拉起子代理。
5. 上下文传递：子代理启动时必须携带相关代码上下文和目标说明。
6. 技能验收：根据任务类型匹配 `skills`，执行验收并返回通过/失败与原因。
7. 结果回写：验收结果与日志回写到 `json`/执行记录。

## 5. 验收标准

1. 能成功读取并遵循 `AGENT` 规范文档。
2. `json` 字段完整且可被正确解析。
3. 子代理能收到完整上下文并完成至少 1 个任务。
4. `skills` 验收可执行，能输出明确结论。
5. 全链路完成后，任务状态与依赖关系一致、无冲突。

## 6. 非功能要求

1. 可观测性：记录每一步输入、输出、耗时、错误。
2. 可恢复性：失败任务支持重试，且不破坏已完成状态。
3. 一致性：并发执行下不出现重复执行/状态覆盖错误。

## 7. 第一轮实验建议

1. 先做 3 个任务（拆分、执行、验收）的最小闭环。
2. 先用 1 个子代理验证上下文传递。
3. 验收只接 1 个 `skill`，跑通后再扩展多技能。
