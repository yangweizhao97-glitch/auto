# 三仓提炼后的借鉴说明

日期：2026-04-13

## 总原则

这套工作流不是把三个仓库拼在一起，而是只提炼对你当前目标最有帮助的部分：

1. 主 AGENT 统一拆任务和派工
2. 子 AGENT 处理独立开发任务
3. 测试 AGENT 做测试和回传
4. 主 AGENT 做最终验收和总结

## 来自 `karpathy/autoresearch`

提炼点：

1. 先有明确任务说明，再进入执行循环
2. 结果必须可验证，不能只看“看起来做完了”
3. 每轮都要做 keep / retry / discard 式决策
4. 失败后不是直接结束，而是回到优化回路

落到当前工作流里：

1. `task_001` 先做 intake 和 baseline
2. `task_004` 测试 AGENT 返回可验证结果
3. `task_005` 主 AGENT 做 accept / retry / block
4. 不通过就回到优化，而不是直接总结

## 来自 `msitarzewski/agency-agents`

提炼点：

1. 工作流由一个 orchestration 角色主导
2. 专项 agent 负责具体任务，不是所有事都由一个 agent 做
3. 每个 agent 必须有明确 deliverable
4. handoff 要有清晰边界和上下文

落到当前工作流里：

1. `main_agent` 负责拆任务和派工
2. `child_agent` 负责单一开发子任务
3. `tester_agent` 负责测试回传
4. `prompts/` 中为不同角色准备了固定 handoff 模板

## 来自 `affaan-m/everything-claude-code`

提炼点：

1. `.codex/` 风格的工作流资产组织方式
2. verification-loop / quality-gate 这种“验证优先”的工作流意识
3. 将角色提示、质量门、执行规则拆成可复用资产
4. 让 Codex 以流程资产而不是临时 prompt 的方式工作

落到当前工作流里：

1. `.codex/`、`prompts/`、`scripts/` 分层
2. `Invoke-QualityGate.ps1` 统一跑检查
3. `tester-handoff.md` 和 `worker-handoff.md` 固定化
4. `tasks.json` 作为整个流程的状态中心

## 最终收敛结果

三仓提炼后，当前工作流只保留一个统一模式：

1. 主 AGENT 拆任务和派工
2. 子 AGENT 开发
3. 测试 AGENT 测试
4. 主 AGENT 验收
5. 通过后总结，不通过则回到优化
6. 用户可以持续追加新需求，主 AGENT 继续派工

这就是目前工作流的最终形态。
